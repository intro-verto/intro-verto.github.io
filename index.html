Шпаргалка для собеседования QA

Первая неделя

Это отклонение фактического результата работы программы от ожидаемого.
Это проверка соответствия между реальным и ожидаемым поведением программы, осуществляемая на конечном наборе тестов, выбранном определенным образом и включает в себя:
  • планирование работ (Test Management)
  • проектирование тестов (Test Design)
  • выполнение тестирования (Test Execution)
  • анализ полученных результатов (Test Analysis)
Между ними есть тонкие различия: дефект — это несоответствие приложения требованиям или ожиданиям, а баг — это отклонение фактического результата от ожидаемого.
QA-тестировщик отвечает за предотвращение ошибок: создает план тестирования, разрабатывает тестовые сценарии и взаимодействует с командой разработки, чтобы гарантировать соответствие продукта требованиям заказчика.
Тестировщик занимается поиском ошибок в продукте, а QA-инженер отвечает за весь процесс обеспечения качества, включая тестирование, анализ, улучшение процессов разработки и предотвращение дефектов.
Выгоднее всего находить и устранять баги на этапе проектирования и анализа требований, так как исправления на поздних этапах дороже из-за перепроектирования, переработки кода и тестирования.
QA-инженеру платят за предотвращение дефектов, экономию времени команды, оптимизацию процессов разработки и защиту бизнеса от убытков, вызванных багами.
SDLC (Software Development Life Cycle) – жизненный цикл программного обеспечения, включающий следующие этапы:
  • Requirement Analysis — Сбор и анализ требований
  • Planning — Планирование процесса разработки
  • Architectural Design — Проектирование архитектуры системы
  • Software Development — Написание кода
  • Testing — Тестирование ПО
  • Deployment — Внедрение готового продукта
Основные методологии разработки ПО:
  • Waterfall (Каскадная модель) – этапы выполняются последовательно. Подходит для простых и стабильных проектов с четкими требованиями.
  • Incremental (Инкрементная модель) – продукт создается по частям (инкрементам). Подходит для проектов с изменяющимися требованиями.
  • Iterative (Итеративная модель) – процесс повторяется с учетом обратной связи. Эффективна для сложных и динамичных проектов.
  • Agile (Гибкая методология) – включает Scrum, Kanban и другие подходы. Идеальна для быстро меняющихся проектов.
  • Spiral (Спиральная модель) – сочетает каскадный и итеративный подходы с акцентом на управление рисками. Подходит для больших и сложных проектов.
  • V-Model (V-образная модель) – акцент на тестировании на каждом этапе. Применяется при четких требованиях.
  • RAD (Rapid Application Development) – быстрые прототипы и минимальная сложность. Подходит для проектов с меняющимися требованиями.
Scrum - это способ организации процессов в компании, основанный на командной работе, адаптивности и непрерывном самосовершенствовании. Основные ритуалы:
  • Планирование — определение целей и задач спринта
  • Ежедневный стендап — короткие встречи команды для синхронизации
  • Обзор спринта — демонстрация результата
  • Ретроспектива спринта — анализ процесса и улучшение командной работы
Kanban - это методология управления проектами для визуализации работы и повышения эффективности. Реализуется через:
  • Kanban-доска — визуализация работы в колонках (В бэклоге, В работе, На проверке, Сделано)
  • Карточки задач — представление работы на доске
  • WIP-лимиты — ограничения на количество задач в каждой колонке
  • Блокеры (опционально) — обстоятельства, мешающие прохождению задачи
Основные стадии жизненного цикла бага:
  • New — зарегистрирован
  • Assigned — назначен на исполнителя
  • Open — подтвержден и взят в работу
  • Fixed — исправлен разработчиком
  • Pending Retest — ожидает повторной проверки
  • Retest — повторно проверяется тестировщиком
  • Reopened — открыт заново, если баг не исправлен (опционально)
  • Verified — подтверждено исправление
  • Closed — закрыт
Дополнительные статусы:
  • Rejected — отклонен, если баг некорректен
  • Duplicate — дубликат существующего бага
  • Deferred — отложен на будущие релизы
  • Not a Bug — не влияет на функциональность
  • Won't Fix — не будет исправлен
  • Couldn't Reproduce — не удалось воспроизвести
  • Need More Information — требуется дополнительная информация
Это модель, где клиент запрашивает данные или услуги, а сервер предоставляет их, обрабатывая запросы. Клиент выступает инициатором, а сервер — обработчиком.
  • Клиент — отправляет запросы серверу и получает результаты
  • Сервер — обрабатывает запросы клиента
  • База данных — хранит и предоставляет данные по запросу сервера
Да, это возможно и часто необходимо для распределения нагрузки, повышения масштабируемости и надежности приложения.
Балансировщик нагрузки — это компонент, который равномерно распределяет входящие запросы между серверами или БД для оптимизации работы системы.
  • Тонкий клиент выполняет минимум операций, полагаясь на сервер (например, веб-браузер, мобильное приложение)
  • Толстый клиент обрабатывает больше данных и логики локально, уменьшая нагрузку на сервер (например, Photoshop или игра Diablo)
Примеры:
  • Тонкий клиент: CRM Salesforce
  • Толстый клиент: Photoshop, Diablo
REST (Representational State Transfer) — это архитектурный стиль для разработки веб-сервисов, использующий HTTP-протокол и методы (GET, POST, PUT, DELETE) для взаимодействия с ресурсами, представленными в виде URL. Основные принципы:
  • Отделение клиента от сервера (Client-Server) — разделение клиентской и серверной логики
  • Отсутствие записи состояния клиента (Stateless) — каждый запрос независим
  • Кэшируемость (Cacheable) — ответы могут быть закешированы
  • Единообразие интерфейса (Uniform Interface) — стандартизированные методы
  • Многоуровневость системы (Layered System) — возможность использования промежуточных слоев
  • Предоставление кода по запросу (Code on Demand) — опциональная передача исполняемого кода
В REST данные могут передаваться в форматах JSON, Protobuf, Plain text, HTML или XML, но JSON является наиболее популярным и предпочтительным.
Главное отличие в том, что REST - это архитектурный стиль, а SOAP - это протокол.
SOAP (Simple Object Access Protocol) — это протокол для обмена структурированными сообщениями между приложениями по сети, основанный на XML. Он не имеет ограничений на тип используемого транспортного протокола, но требует больше пропускной способности и ресурсов, чем REST.
В SOAP данные передаются в формате XML, а для описания сервиса используется WSDL (Web Services Description Language) — XML-язык для описания веб-сервисов и доступа к ним.
Это архитектура, где вся система представляет собой единое приложение, где все компоненты связаны друг с другом и работают как единое целое.

Плюсы:
  • Простота разработки и развёртывания
  • Легче тестировать и отлаживать
  • Единое хранилище кода
Минусы:
  • Трудности масштабирования
  • Сложность внесения изменений
  • Уязвимость: ошибка в одной части может нарушить всю систему
Это архитектура, где система разделена на отдельные независимые модули (микросервисы), которые взаимодействуют через API.

Плюсы:
  • Легче масштабировать
  • Возможность использования разных технологий для микросервисов
  • Повышенная устойчивость: сбой одного сервиса не влияет на остальные
Минусы:
  • Более сложная разработка и поддержка
  • Сложная система управления и взаимодействия между сервисами
  • Увеличение задержек сетевых запросов между сервисами
Существует два основных способа взаимодействия:
Синхронные способы общения:
  • Синхронный REST-like и аналоги
  • gRPC
  • SOAP
Асинхронные способы общения:
  • Месседжинг - RabbitMQ, ZeroMQ, ActiveMQ
  • Стриминг - Kafka

Вторая неделя

STLC (Software Testing Life Cycle) — это процесс, включающий этапы планирования, разработки, выполнения и завершения тестирования программного обеспечения.

Основные этапы:
  • Анализ требований — изучение требований для подготовки к тестированию
  • Планирование тестирования — определение стратегии, ресурсов, сроков и метрик
  • Проектирование тестов — разработка тест-кейсов и подготовка тестовых данных
  • Настройка среды тестирования — подготовка инструментов и инфраструктуры
  • Выполнение тестов — запуск тест-кейсов, фиксирование дефектов
  • Закрытие тестирования — анализ результатов, подготовка отчетов и завершение процесса
7 основных принципов тестирования:
  • Исчерпывающее тестирование невозможно — протестировать все возможные сценарии нереально, важно сосредоточиться на приоритетных
  • Тестирование демонстрирует наличие дефектов, а не их отсутствие — тесты подтверждают наличие ошибок, но не гарантируют, что их нет
  • Заблуждение об отсутствии ошибок — отсутствие найденных багов не означает, что продукт идеален или готов
  • Раннее тестирование сохраняет время и деньги — чем раньше найдены ошибки, тем дешевле и быстрее их исправить
  • Принцип скопления дефектов — большинство багов концентрируются в ограниченном числе модулей
  • Тестирование зависит от контекста — подходы различаются в зависимости от типа и целей продукта, команды и сроков
  • Парадокс пестицида — повторяющиеся тесты теряют эффективность, их нужно регулярно обновлять
Требования — это описания того, что система должна выполнять для достижения целей пользователя или бизнеса.

Атрибуты требований:
  • Полнота — все аспекты задачи учтены, ничего не упущено
  • Однозначность — формулировки четкие, исключают двойное толкование
  • Непротиворечивость — требования согласованы между собой, нет конфликтов
  • Необходимость — включено только то, что реально нужно для цели проекта
  • Осуществимость — можно реализовать с учетом доступных ресурсов и технологий
  • Тестируемость — выполнение требования можно проверить однозначно
Основные метрики тестирования:
  • Passed/Failed Test Cases - соотношение успешных и неуспешных тестов
  • Not Run Test Cases - количество невыполненных тестов
  • Open/Closed Bugs - соотношение открытых и закрытых багов
  • Reopened/Closed Bugs - соотношение переоткрытых и закрытых багов
  • Bugs by Severity/Priority - количество багов по серьезности/приоритету
  • Покрытие тестовыми сценариями и чек-листами
  • Процент написанных тестовых сценариев
  • Метрики выполнения тестов (время прохождения, общая статистика)
  • Метрики дефектов (плотность, количество, частота отказов)
  • Информация о загруженности команды и трудозатратах
  • Финансовые метрики
Метрики тестирования нужны для:
  • Отслеживания прогресса команды по срокам проекта и дедлайнам
  • Качественной оценки текущего состояния системы
  • Контроля качества процесса тестирования
  • Постановки целей и эффективного планирования на основе понимания существующих проблем
  • Визуального представления результатов для выявления узких мест
  • Численного выражения достижения заданного уровня качества
